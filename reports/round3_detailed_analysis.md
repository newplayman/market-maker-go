## 💡 下一步优化建议

### 核心问题诊断

当前策略的核心矛盾:
1. **价差太小(6bps)无法覆盖成本** → 订单PnL为负
2. **价差增大会进一步减少成交** → 平手订单已经55.9%
3. **风控太严(netMax=0.06)限制盈利** → 无法通过方向获利

### 建议调整方向

#### 方案A: 回归Round 2参数,但增强Maker比例
```yaml
minSpread: 0.0004          # 回到4bps,增加成交机会
quoteIntervalMs: 500       # 回到500ms,保持稳定性
netMax: 0.10               # 回到0.10,允许更大盈利空间
staticFraction: 0.9        # 从0.8提升到0.9,进一步减少Taker
```

**预期效果**:
- 订单PnL恢复正值(参考Round 2的+32 USDC)
- 但需要严格监控Taker占比和滑点

#### 方案B: 激进定价,减少订单数量
```yaml
minSpread: 0.0010          # 提升到10bps,确保每单盈利
quoteIntervalMs: 1000      # 降低频率到1秒,减少无效订单
netMax: 0.08               # 适度放宽到0.08
staticFraction: 0.85       # 保持85%静态挂单
```

**预期效果**:
- 大幅减少平手订单(从55.9%降到30%以下)
- 提升盈利订单占比(从22.2%到35%+)
- 但成交频率会下降

#### 方案C: 混合策略 (推荐)
```yaml
minSpread: 0.0005          # 5bps,在4bps和6bps之间
quoteIntervalMs: 400       # 400ms,在300ms和500ms之间
netMax: 0.08               # 0.08,在0.06和0.10之间
singleMax: 0.032           # 提升到0.032,加快减仓速度
staticFraction: 0.85       # 85%静态挂单
reduceOnlyThreshold: 15.0  # 从20.0降到15.0,适度收紧风控
```

**预期效果**:
- 平衡成交频率和订单质量
- 预期盈利订单占比30%+
- 预期平手订单占比40%以下
- 预期30分钟净盈利 +0.5~1.0 USDC
